Architecture Approach Comparison

Three ways to implement the 3-layer routing

A

Route Groups + Shared Components

Three Next.js route groups: (superadmin)/, (senior)/, (user)/. Each has its own layout.tsx with layer-specific guard and sidebar. Pages import shared components from components/ but compose them differently per layer. Middleware reads role cookie and redirects to the correct layer.

Pros

  • Clear physical separation in URL and filesystem
  • Middleware can enforce access at the edge
  • Each layer's layout is independent
  • Easy to add layer-specific pages later

Cons

  • More files (3 layouts, 3 sidebar configs)
  • Some pages appear in multiple layers (Dashboard, Settlement)
B

Single Route Group + Dynamic Layout

Keep one (dashboard) route group. Layout reads role from cookie and dynamically renders the appropriate sidebar and guard. URLs stay flat (/dashboard, /user-management). Middleware only checks auth, not role-based routing.

Pros

  • Fewer files, single layout
  • Pages only exist once
  • Simpler migration from current structure

Cons

  • No URL-level separation — User can't tell they're in /user vs /senior
  • Doesn't match the spec's /superadmin/*, /senior/*, /user/* routing
  • Client-only role guard is less secure than middleware
  • Can't easily restrict which pages exist per layer at the routing level
C

Hybrid: Route Groups + Smart Redirects

Same 3 route groups as Approach A, but pages that exist in multiple layers (Dashboard, Settlement, Bank APIs) are defined once in a components/ directory and imported into each layer's page. Middleware reads role cookie, enforces access, and redirects /dashboard → /{role}/dashboard so users always land in the right layer.

Pros

  • Best of both: URL separation + shared components
  • Middleware enforces at the edge
  • Smart redirect means /dashboard auto-routes to /{role}/dashboard
  • Components are truly DRY — one StatementTable, one NetBalanceChart

Cons

  • Slightly more complex file structure
  • Need to maintain layer-specific page.tsx files that import shared components
